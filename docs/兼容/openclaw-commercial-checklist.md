# ThousandCliffs-AI OpenClaw 商用适配清单

> 面向当前阶段的最高优先级目标：先打通商用业务链路，支撑联调、测试与上线准备。
>
> 定位说明：
> - Web 业务后端仍是站点主中枢，负责订单、工单、项目、支付、通知、WebSocket 等业务域。
> - 当前 SAVC / OpenClaw 仓库承接“云上 Agent 能力层”。
> - 本地 LLM 仍保持为独立服务，本轮不与云上 OpenClaw 合并实现。
> - 学习、反思、成熟化收益保留为次级目标，不抢占商用主线资源。

## 1. 目标范围

本清单只覆盖“站点业务后端 <-> 云上 OpenClaw / SAVC”这条链路，不覆盖：

- Web 前端已有页面开发
- FastAPI 业务后端已有通用模块开发
- 本地 LLM 模型服务本体实现
- 媛媛长期学习系统的完整建设

当前阶段的唯一主目标是：

```text
业务后端能够稳定调用云上 OpenClaw 服务完成
任务创建 -> 状态查询 -> 消息交互 -> 进度回传 -> 完成回调
并具备最小可运维、可排障、可人工兜底能力
```

## 2. 系统边界

### 2.1 业务链路位置

```text
闲鱼 / Web 站点订单
  -> Web 业务后端
  -> 本地 LLM（需求分析 / 分类）
  -> 云上 SAVC / OpenClaw（规划 / 执行 / 沟通 / 进度）
  -> Web 业务后端
  -> Web 前端 / 管理后台 / 通知
```

### 2.2 当前阶段的职责切分

| 模块 | 职责 |
|------|------|
| Web 业务后端 | 项目主数据、订单和项目状态机、用户权限、支付、通知、站点 WebSocket |
| 本地 LLM | 需求分类、复杂度评估、结构化摘要 |
| SAVC / OpenClaw | 执行任务、规划任务、项目沟通、生成进度结果、输出结构化任务结果 |
| 人工管理员 | 审核、兜底、交付前人工把关 |

### 2.3 本轮不做的事

- 不把订单、支付、卡密等主业务数据迁入媛媛记忆系统
- 不把业务后端职责改由 SAVC 仓库承担
- 不优先做复杂多节点调度、自动学习、复杂自省工作流
- 不把本地 LLM 并进云上 OpenClaw 服务层

## 3. P0：联调前必须完成

> 这些项不完成，后续“测试上线”没有稳定基础。

### 3.1 服务入口与鉴权

- [ ] 提供面向 Web 业务后端的独立任务服务入口
- [ ] 提供健康检查接口，例如 `GET /health`
- [ ] 提供服务鉴权机制，例如内部 API Key / HMAC / allowlist
- [ ] 明确区分“外部业务调用接口”和“内部调试接口”
- [ ] 禁止业务后端直接依赖 `savc_*` 工具调用细节

### 3.2 任务接口最小集合

- [ ] `POST /tasks`
  - 用于创建任务
  - 至少支持：
    - `requirement_analysis`
    - `development_planning`
    - `development_execution`
- [ ] `GET /tasks/:id`
  - 返回任务状态、摘要、错误信息、时间戳
- [ ] `POST /tasks/:id/messages`
  - 用于需求沟通阶段继续给项目对应 Agent 发消息
- [ ] `POST /tasks/:id/cancel`
  - 至少支持软取消或标记取消
- [ ] `GET /tasks/:id/events`
  - 返回时间线事件，便于联调和排障

### 3.3 任务状态机

- [ ] 定义统一任务状态：
  - `accepted`
  - `queued`
  - `running`
  - `waiting_user`
  - `completed`
  - `failed`
  - `canceled`
  - `review_required`
- [ ] 定义状态推进规则，避免同一任务出现非法跳转
- [ ] 定义“项目状态”和“OpenClaw 任务状态”的映射关系
- [ ] 允许业务后端基于状态安全重试或人工接管

### 3.4 任务类型映射

- [ ] 明确 `requirement_analysis` 在当前 SAVC 中由哪类 agent / workflow 承担
- [ ] 明确 `development_planning` 的产出结构
- [ ] 明确 `development_execution` 的执行边界
- [ ] 明确“用户沟通消息”走哪个会话、如何续聊、如何保持上下文
- [ ] 明确“单轮项目任务”和“持续项目会话”的绑定方式

### 3.5 回调 / Webhook 闭环

- [ ] 支持向业务后端回调以下事件：
  - [ ] `analysis`
  - [ ] `plan`
  - [ ] `progress`
  - [ ] `complete`
  - [ ] `message`
- [ ] 每类回调都必须有固定 JSON 契约
- [ ] 回调必须带签名或内部鉴权头
- [ ] 回调失败必须记录，并支持重试
- [ ] 回调必须支持幂等

### 3.6 任务持久化与排障

- [ ] 任务元数据必须持久化，不能只存在内存
- [ ] 保存任务与项目的关联关系：
  - `taskId`
  - `projectId`
  - `taskType`
  - `sessionKey`
  - `runId`
  - `childSessionKey`
- [ ] 保存最小事件流：
  - 创建
  - 状态变更
  - 回调投递
  - 人工修正
  - 失败原因
- [ ] 记录请求 ID / trace ID，便于业务后端与 SAVC 双边排障

### 3.7 人工兜底能力

- [ ] 任务失败后可标记为 `review_required`
- [ ] 支持人工补发回调
- [ ] 支持人工追加项目消息
- [ ] 支持人工关闭任务
- [ ] 支持人工写入“修正结论”，便于后续复盘

### 3.8 最小安全与脱敏

- [ ] 不把支付、卡密、账号密码、明文令牌写入长期记忆
- [ ] 任务日志中对手机号、邮箱、订单号、支付字段做脱敏
- [ ] 站点业务后端与 OpenClaw 服务之间使用独立密钥
- [ ] 区分调试日志和业务日志，避免敏感数据泛滥

### 3.9 联调与测试

- [ ] 提供本地 / 测试环境用的假任务回调模式
- [ ] 提供最小联调脚本或 Postman / curl 示例
- [ ] 提供任务创建 -> 状态查询 -> 回调投递的 smoke test
- [ ] 至少准备 3 类业务样本：
  - [ ] 需求清晰样本
  - [ ] 需求模糊样本
  - [ ] 中途多轮补充需求样本
- [ ] 至少准备 3 类异常样本：
  - [ ] 回调失败
  - [ ] 任务超时
  - [ ] Agent 输出结构不合法

## 4. P1：上线前建议完成

> 这些项不一定阻塞第一轮联调，但会明显影响上线稳定性。

### 4.1 可靠性增强

- [ ] 回调重试队列
- [ ] 死信队列 / 失败任务池
- [ ] 幂等键支持
- [ ] 超时任务扫描
- [ ] 任务恢复 / 重放能力

### 4.2 运维能力

- [ ] 任务列表查询接口
- [ ] 按 `projectId` / `taskType` / `status` 检索
- [ ] 最近失败任务视图
- [ ] 回调失败视图
- [ ] 手动重试按钮或脚本入口

### 4.3 质量约束

- [ ] 对 `analysis` / `plan` / `progress` 回调做 schema 校验
- [ ] 对 Agent 结果做统一格式化和容错处理
- [ ] 定义可接受的最大执行时长
- [ ] 定义人工介入阈值

### 4.4 最小学习闭环

- [ ] 保存任务输入摘要
- [ ] 保存任务输出摘要
- [ ] 保存失败原因分类
- [ ] 保存人工修正记录
- [ ] 仅把脱敏后的流程经验写入程序记忆

## 5. P2：上线后再做

> 这部分有价值，但不应该压过当前业务打通优先级。

- [ ] 多节点 / 多网关调度策略
- [ ] 自动路由到不同 agent 组合
- [ ] 自动提示词优化
- [ ] 自动失败归因聚类
- [ ] 从业务任务自动提炼 workflow 模板
- [ ] 商用任务对媛媛人格和反思系统的长期反馈策略
- [ ] 更完整的自学习、自我评估、自我升级链路

## 6. 验收口径

### 6.1 联调可用标准

- [ ] 业务后端可以成功创建任务
- [ ] 可以查询任务状态
- [ ] 可以继续发送项目沟通消息
- [ ] OpenClaw 侧可以回调业务后端
- [ ] 业务后端可以根据回调更新项目状态
- [ ] 整条链路至少完成一次端到端 smoke

### 6.2 测试上线标准

- [ ] 需求分析任务连续通过率达到可接受水平
- [ ] 回调失败可重试
- [ ] 任务失败可人工接管
- [ ] 日志可定位到单次任务全链路
- [ ] 关键敏感信息完成脱敏
- [ ] 测试环境可复现至少 5 个典型项目样本

### 6.3 第一轮上线标准

- [ ] 仅开放有限任务类型
- [ ] 仅允许白名单项目进入自动链路
- [ ] 关键节点保留人工审核
- [ ] 回调异常有告警
- [ ] 有明确回滚方案
