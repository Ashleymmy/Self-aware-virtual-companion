## 2) Docker 部署 SAVC 开发容器并完成启动排障

- 对话目标
  - 先了解当前项目结构，再用 Docker 将项目实际启动起来，并把 `savc-gateway` / `savc-ui` 都跑通。

- 执行内容
  - 检查仓库的 Docker 编排、启动脚本、插件依赖和 OpenClaw 集成方式，确认 `openclaw/` vendored 快照不完整，无法作为镜像内构建来源稳定使用。
  - 将 `infra/docker/docker-compose.cloud.yml` 切换为直接使用 `node:22-bookworm` 运行时容器，并在容器启动时执行 `pnpm --dir /workspace install --no-frozen-lockfile --lockfile=false`。
  - 将容器内 `OPENCLAW_ROOT` 改为 pnpm 隔离布局下稳定可用的 `/workspace/node_modules/.pnpm/node_modules/openclaw`，同时修正 Compose 命令块中的变量转义，避免宿主机侧提前展开成 `/srv/devrepo/.../openclaw`。
  - 将 gateway 健康检查从 `/health` 调整为 `/healthz`。
  - 修复 `scripts/setup.sh` 生成的 `openclaw.json`：
    - 去掉会导致启动期强制 secret 解析失败的 provider `apiKey` 占位写法
    - 为当前 Docker/LAN 场景增加 `SAVC_GATEWAY_CONTROL_UI_HOST_FALLBACK` 开关，并在容器部署中默认开启 Host-header fallback，绕过非 loopback Control UI 的强制 origin 白名单门槛
  - 为 `packages/plugin` 增加构建配置，生成 `dist/`，并把 OpenClaw 插件入口从 `index.ts` 改为 `dist/index.js`，避免运行时直接加载 TypeScript 源。
  - 为 gateway 容器增加 `NODE_OPTIONS=--max-old-space-size=1536`，让 OpenClaw 在当前 2G 内存机器上跨过启动阶段的堆峰值。
  - 多次顺序执行 `bash scripts/infra/dev_container.sh down` / `up`，结合 `docker logs`、`docker inspect`、`docker exec` 逐步排除入口路径、缺失 secret、Control UI 安全策略和 V8 OOM 等问题，最终完成服务启动。

  - `docker exec savc-cloud-savc-gateway-1 node -e "fetch('http://127.0.0.1:18789/healthz')..."`：通过，返回 `200` 和 `{"ok":true,"status":"live"}`。
  - `docker exec savc-cloud-savc-ui-1 node -e "fetch('http://127.0.0.1:5174/')..."`：通过，返回 `200`，首页 HTML 正常输出。
  - `docker inspect savc-cloud-savc-gateway-1 --format '{{json .Config.Env}}'`：通过，确认容器内已注入 `NODE_OPTIONS=--max-old-space-size=1536`。
  - `docker logs savc-cloud-savc-gateway-1 --tail ...`：通过，确认插件入口已从 `packages/plugin/index.ts` 切换为 `packages/plugin/dist/index.js`，并且在最终成功启动后未再出现缺失 secret、Control UI origin 拒绝或 V8 OOM 崩溃。
